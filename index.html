Purpose

As front-end devs at Envy Labs, we work primarily in medium-to-large-sized Rails applications. Each of these projects has a sizable amount of CSS undergoing normal maintenance and iteration. Depending on complexity and setup, changes to existing style can be difficult or done poorly if the right way is too convoluted.

MVCSS (MVC-based Style Sheets) is our method for creating predictable and maintainable application style by coupling Rails controller and controller action naming conventions to contextual style tweaks.

What? Another Way to Write CSS?

We're big fans (and often employ tenets of) well-documented methods like OOCSS, BEM, and SMACSS. These setups tend to get us ~90% through a build flawlessly before taking an unwanted turn - usually as a result of organization issues in the face of placement-dependent alterations (more on that in the Layout section).

All This for CSS?

CSS in and of itself is simple to pick up, but difficult to apply maintainably to large applications - hence the popularity of the methods listed above. MVCSS is one such manner to organize styles, and the one that we've had the most success in employing.

Do I Have to Use Sass?

Sass is definitely not required to employ this paradigm, but we're big fans (hence the default project files being in SCSS - blame Rails). Most of the Sass-only stuff is found in Settings.

Organizational Overview

MVCSS is broken down into three primary sections - Global (Resets, Settings, and Defaults), Modules (self-contained, reusable chunks), and Layout (context-based tweaks to base and module styles).

Cascade Down, Abstract Up

It's important to remember that declarations can always be moved and abstracted. If you find you're repeating position-determined styles in the Layout section, it's probably time to abstract those styles to a new or existing module. Alternatively, if you're declaring too many modular tweaks just for one-off instances, it might be time to move those tweaks to the Layout section.

Global

Reset / Normalize

As long as the project size warrants it, we always get things started with a style reset. Eric Meyer's Reset CSS is still our favorite, but alternatives like Normalize.css are easily interchangeable.

Settings

The settings partial centers around reusable utilities - @font-face declarations, Sass mixins and variables, classes to be used with @extend (soon to be extend directives), and keyframe animations. Grouping everything into one file allows newcomers to the project the means to find out what's available to them quickly.

A Note About Variables

We won't be delving too far into our CSS style guides for now, but we definitely recommend two things when making use of Sass variables: prefix the variable with what it represents ($color-base, $color-highlight, $font-base) and make use of semantic names ($color-base rather than $color-blue).

Base

Base contains declarations for block-level text items, inline tags, and any commonly occurring content containers. Text styles set here reflect the most common appearance of that tag in the design.

Modules

Modules in this context refer to any unit of style that can be found across multiple pages. Most of your style should be found here. This draws from the OOCSS idea of separating everything into usable chunks, making it second-nature if you're familiar with that method.

Naming

While the decision between purely semantic and presentational class names is up to you, we definitely recommend naming modules independently of their possible placement - .search might appear on the top of one page and the bottom of another.

Examples of Modules

Here are some commonly occurring examples of modules:

  • Modals
  • Buttons
  • Tooltips
  • Navigation
  • Re-appearing chunks of content, like a user info card or sidebar

Layouts

Having resets, base styles, and style modules is definitely not new - so what's the point of this MVCSS stuff? Inevitably, designs you receive (or design yourself) will contain small tweaks of base styles depending on where the element sits. For example, a sidebar on a user settings page might have a border around it, or the headline on a home page could suddenly turn white. With trump cards like balance and contrast, we're usually on the rope to figure out how to incorporate the designer's vision without messing up our precious order.

Note: again, if you find yourself repeating tweaks across multiple layouts, it's time to abstract up to the module level.

Context-Based Module Adjustment

The primary idea behind MVCSS is that you can write your HTML after setting up the Global and Module sections and employ the Layout section afterwards for any minor fixes.

Controllers / Controller-Action

To make these contextual tweaks, we've found it easiest to couple the sections to Rails controllers. Outputting the controller name (such as Users) and the action name (such as Edit) in automatically to the body is trivial in Rails, giving us the ability to tweak modules as such:

/* Modules */

.btn {
  background: $color-base;
  color: darken($color-base, 20%);
  display: block;
  padding: 5px;
}

/* Layout */

.users { /* Controller name */
  .btn {
    color: #fff; /* Controller-wide tweaks */
  }
}

.users-show { /* Controller action */
  .btn {
    color: $color-highlight; /* Tweaks only found on a particular action */
  }
}

Object Notes

As mentioned before, feel free to employ whatever you're comfortable with in regards to module naming, semantics, and abstraction. We tend to only use classes, but aren't as strict about using descendant selectors (.tooltip h4) as OOCSS recommends against. It's best to get the properties into the stylesheet without fretting over the proper level of abstraction initially, then refactoring as you continue to build out the application.
